# Campus Platform 自动化测试总结报告

## 1. 文档说明

本文档用于说明 Campus Platform 项目的自动化测试设计、测试范围、测试用例覆盖内容、执行结果、问题原因分析和后续改进方向。测试对象为已部署的校园平台前后端服务，测试方式以黑盒自动化测试为主，通过测试端 Web Runner 一键触发，也支持按测试类型单独执行。

本次全量测试共收集 `127` 条自动化用例，最终结果为 `126 passed, 1 skipped`，整体通过率为 `99.2%`。唯一跳过项为教师课表接口在当前后端环境中返回受控业务异常，未影响测试套件整体执行。

## 3. 测试目标

本次测试的目标不是只验证某一个接口能否访问，而是从交付验收角度确认系统是否具备稳定上线条件，主要关注以下方面：

- 前端页面是否可访问，刷新深层路由是否正常。
- 后端 API 是否可访问，统一响应格式是否稳定。
- 登录态接口、学生、教师、管理员角色接口是否可用。
- 业务模块分页、列表和只读接口是否符合前端消费要求。
- 权限边界是否正确，匿名用户和低权限用户是否被拦截。
- 异常输入、错误 token、错误方法是否受控，不应导致服务崩溃或泄露堆栈。
- 生产部署配置是否合理，包括静态资源、CORS、API 错误页和 SPA fallback。
- 安全基线是否满足上线要求，敏感路径不应公开。
- 核心页面和接口是否具备基础响应性能与小并发稳定性。

## 5. 测试用例覆盖说明

### 5.1 冒烟测试

冒烟测试用于部署完成后的第一轮快速验证，确认系统处于可访问状态。

覆盖内容：

- 访问前端首页 `/`，检查 HTTP 状态码、`Content-Type` 和 Vue 应用挂载节点。
- 调用后端验证码接口 `/auth/captcha`，检查统一响应格式、`captchaKey` 和 `captchaImage`。
- 访问核心前端路由：`/login`、`/dashboard`、`/campus/notice`、`/campus/book`、`/education/timetable`。
- 调用核心公开 API：验证码、公告分页、图书分页、宿舍楼列表、宿舍房间列表。

判断标准：

- 页面返回 HTML 应用壳。
- API 返回 `{code,msg,data}` 结构。
- 不出现 HTTP 5xx 服务端错误。

### 5.2 黑盒部署测试

黑盒部署测试从公网用户视角验证系统，不依赖服务器内部日志、数据库或源码。

覆盖内容：

- 前端根路径、登录页、工作台、公告、课程、用户管理等深层路由刷新。
- 不存在的前端路由是否仍由 SPA fallback 返回应用壳。
- 不存在的 API 路径不能返回前端 `index.html`。
- 前端首页必须引用生产构建后的 JS/CSS 静态资源。
- 后端 API 前缀在公网环境下可访问，返回 JSON 而不是 HTML。

判断标准：

- 前端深层路由刷新不 404。
- API 错误响应不被 Nginx 错误转发到前端页面。
- 生产构建资源存在并可被浏览器加载。

### 5.3 部署质量测试

部署质量测试聚焦上线配置是否规范。

覆盖内容：

- 前端首页是否包含基础可访问性元信息，例如 `lang`、`viewport`、`title`。
- JS/CSS 静态资源是否返回非 HTML 内容，避免资源请求被错误 fallback。
- CORS 预检请求是否受控，不产生服务端 500。
- 缺失 API 路径是否不会泄露前端页面或 Java/Spring 堆栈。

判断标准：

- 资源 Content-Type 合理。
- API 错误页受控。
- 部署层没有明显反向代理错误。

### 5.4 前端页面测试

前端页面测试用于巡检主要业务页面是否能正常打开。

覆盖路由：

- 认证相关：`/login`、`/register`。
- 工作台：`/dashboard`。
- 系统管理：`/system/user`、`/system/role`、`/system/menu`、`/system/dict`。
- 教学管理：`/education/course`、`/education/timetable`、`/education/attendance`、`/education/score`、`/education/leave`。
- 校园服务：`/campus/notice`、`/campus/dorm`、`/campus/repair`、`/campus/card`、`/campus/book`、`/campus/message`。
- 统计页面：`/statistics`。

判断标准：

- 路由返回 HTTP 成功状态。
- 返回内容为 HTML。
- 页面包含 Campus Platform 标识或 Vue 应用挂载节点。

### 5.5 接口契约测试

接口契约测试用于确认前后端协作基础是否稳定。

覆盖内容：

- 公告分页 `/svc/notice/page?pageNum=1&pageSize=5`。
- 图书分页 `/svc/book/page?pageNum=1&pageSize=5`。
- 验证码 `/auth/captcha`。
- 所有接口统一检查 `{code,msg,data}` 格式。
- 分页接口检查 `records` 和分页元数据。
- 验证码检查 `captchaKey` 长度和图片内容。
- 非严格模式下不强制时间格式，严格模式下检查是否存在 ISO `T` 格式时间。

判断标准：

- 响应必须是 JSON 对象。
- 必须包含 `code`、`msg` 字段。
- 成功接口 `code` 应为 `200`。
- 分页结构必须满足前端列表渲染需要。

### 5.6 公开接口测试

公开接口测试验证无需登录即可访问或应被受控拒绝的接口。

覆盖内容：

- `/auth/captcha` 验证码接口。
- `/svc/notice/page` 公告分页接口。
- 空登录请求 `/auth/login` 是否被受控拒绝。

判断标准：

- 公开接口成功返回统一结构。
- 如果环境中接口被配置为需要登录，应返回 `401/403`，不能返回 500。
- 空登录请求不能成功，必须返回受控错误信息。

### 5.7 登录态接口测试

登录态接口测试使用测试账号或测试登录旁路获取 token，覆盖学生、教师、管理员三类角色。

公共登录态接口覆盖：

- 当前用户信息 `/auth/userInfo`。
- 工作台概览 `/dashboard/overview`。
- 公告、图书、报修、校园卡、宿舍、消息等个人相关接口。

管理员接口覆盖：

- 用户分页 `/sys/user/page`。
- 角色列表 `/sys/role/list`。
- 菜单树 `/sys/menu/tree`。
- 字典类型 `/sys/dict/type/list`。
- 登录日志 `/sys/log/login`。
- 课程、成绩、报修、校园卡记录、宿舍分配等管理接口。

教师接口覆盖：

- 教师课程分页 `/edu/course/page`。
- 教师课表 `/edu/timetable/my`。
- 活跃考勤 `/edu/attendance/session/active`。
- 请假审批 `/edu/leave/page`。
- 退课申请 `/edu/elective/drop-requests/pending`。
- 教师消息课程 `/sys/message/my-courses`。

学生接口覆盖：

- 学生课表 `/edu/timetable/my`。
- 可选选修课 `/edu/elective/available`。
- 我的选修课 `/edu/elective/my`。
- 我的考勤 `/edu/attendance/my`。
- 我的请假 `/edu/leave/my`。
- 我的评价 `/edu/evaluation/my`。
- 我的教师 `/sys/message/my-teachers`。

判断标准：

- token 有效时接口应返回统一成功结构。
- token 无效或未配置时，应返回登录要求，而不是产生大量失败项。
- 受控业务异常会在报告中明确标记为跳过或环境问题。

### 5.8 业务模块测试

业务模块测试关注核心业务分页结构是否满足前端展示需求。

覆盖接口：

- 用户分页 `/sys/user/page?pageNum=1&pageSize=5`。
- 课程分页 `/edu/course/page?pageNum=1&pageSize=5`。
- 报修分页 `/svc/repair/page?pageNum=1&pageSize=5`。

判断标准：

- 返回统一接口结构。
- `data` 必须是对象。
- 必须包含 `records` 列表。
- 必须包含分页元数据，例如 `total`、`current`、`pageNum` 或 `pageSize`。

### 5.9 权限测试

权限测试用于验证 RBAC 权限边界是否正确。

覆盖内容：

- 匿名用户访问受保护接口：`/auth/userInfo`、`/sys/user/page`、`/sys/role/list`、`/edu/course/page`。
- 学生用户访问管理员接口：用户分页、角色列表、菜单树、登录日志。
- 管理员访问用户分页接口。

判断标准：

- 匿名用户必须被 `401/403` 拒绝。
- 学生访问管理员接口必须被 `401/403` 拒绝。
- 管理员 token 有效时应能访问管理员接口。

### 5.10 负向输入测试

负向输入测试用于验证系统健壮性和基础安全防护。

覆盖内容：

- 非法分页参数：负页码、超大页大小、非数字页码、非数字页大小。
- 错误 token：普通错误字符串、`Bearer bad-token`、UUID 形式错误 token。
- 注入类输入：SQL 注入特征、XSS 脚本、路径穿越、空字节。
- 错误 HTTP 方法：对验证码接口发起错误方法调用。

判断标准：

- 不允许出现 HTTP 5xx。
- 不允许返回 Java/Spring 堆栈。
- 不允许反射危险脚本内容。
- 错误 token 必须被拒绝。

### 5.11 安全基线测试

安全测试用于验证上线环境不暴露敏感能力。

覆盖路径：

- `/actuator/env`。
- `/actuator/heapdump`。
- `/swagger-ui/index.html`。
- `/v3/api-docs`。
- `/../application.yml`。
- `/uploads/../application.yml`。

其他安全检查：

- 不存在 API 路径不应泄露 `java.lang`、`stacktrace` 或 `org.springframework`。
- 前端安全响应头在严格模式下可进一步检查 `X-Content-Type-Options`、`X-Frame-Options`、`Content-Security-Policy`、`Referrer-Policy`。

判断标准：

- 敏感路径不应公开返回成功业务数据。
- 错误响应不应暴露堆栈和框架内部信息。
- 非严格模式下允许 Swagger 类接口作为开发配置存在，但上线建议关闭或加权限。

### 5.12 性能与稳定性测试

性能测试用于观察核心入口是否具备基础响应能力。

覆盖内容：

- 前端首页多轮响应耗时。
- 验证码接口多轮响应耗时。
- 公告分页接口多轮响应耗时。
- 公共 API 小并发访问：验证码、公告、图书、宿舍楼、宿舍房间。
- 前端路由小并发访问：首页、登录页、工作台、公告、课表。

判断标准：

- 默认非严格模式下主要判断服务可用和无 5xx。
- 严格模式下可启用平均耗时和最大耗时阈值。
- 小并发访问不能导致服务端错误。

## 7. 结果分析

本次测试结果为 `126 passed, 1 skipped`，全量执行退出码为 `0`，说明测试套件判定本次验收通过。系统在前端访问、后端接口契约、业务只读接口、权限隔离、负向输入、安全基线和基础性能方面整体表现稳定。

- 后端 API 基本遵循统一响应结构。

- 非法输入不会导致明显服务崩溃或堆栈泄露。

- 核心页面和接口在基础并发下保持可用。

### 7.2 跳过项说明

跳过项：

```text
SKIPPED [1] tests/test_api_authenticated.py:60:
teacher timetable returned controlled backend busy response
```

对应接口：

```text
GET /edu/timetable/my
```

对应角色：

```text
教师账号 ldx3
```

原因分析：

该接口在测试过程中返回了受控业务异常，错误信息为“系统繁忙请求异常，请稍后再试或联系管理员”。测试框架将其识别为后端当前环境或业务数据问题，而不是测试脚本错误，因此标记为 `skipped`。这样可以避免一个后端临时业务异常导致整份验收报告失败，同时在报告中保留问题线索。

可能原因包括：

- 教师账号 `ldx3` 未绑定课程、班级或课表数据。
- 课表查询依赖的学期、课程、教师关联数据不完整。
- 后端接口对空数据场景处理不够友好。
- 查询过程中出现空指针、SQL 条件异常或业务分支未覆盖。

建议修复方向：

- 检查 `ldx3` 教师账号是否存在有效课程和课表安排。
- 后端接口对“无课表”场景应返回 `code=200` 和空列表，而不是业务 500。
- 在后端日志中定位 `/edu/timetable/my` 的异常栈，补充空值保护。
- 增加后端单元测试，覆盖教师无课表、未绑定班级、无当前学期等场景。

### 7.3 安全测试结果说明

安全测试单独执行 `8 passed`，说明当前环境通过了以下检查：

- Actuator 敏感路径未直接公开高风险数据。
- 配置文件路径未被直接读取。
- 错误响应未暴露 Java/Spring 堆栈。
- 前端首页在非严格模式下基础响应正常。

需要注意的是，当前测试默认 `STRICT_MODE=false`，因此 Swagger 和安全响应头采用宽松验收策略。若系统进入生产交付阶段，建议开启严格模式进一步验证：

```bash
bash scripts/run_tests.sh --target http://8.138.10.124 --api http://114.132.77.204:8080/api --profile security --strict
```

严格模式下应重点检查：

- 是否配置 `X-Frame-Options` 或 `Content-Security-Policy`。

## 8. 测试端设计说明

测试端采用 Web Runner 方式提供按钮化操作，降低命令行使用成本。测试人员可以通过浏览器选择单项测试或全量测试。

页面能力包括：

- 一键全量测试。
- 12 类测试独立执行。
- 每类测试独立生成实时页、看板页、总结页。
- 自动记录执行日志和退出码。
- 生成 AI 总结，基于真实测试数据分析通过率、失败项、跳过项和改进建议。

```text

  security/
    latest-live.html

    latest-live.html

    latest-summary.html
```

这种设计保证每个测试类型只展示自己的用例和结果，全量测试展示全部 127 条用例，避免不同页面之间数据混杂。

## 9. 风险与不足

当前测试结果总体良好，但仍存在以下需要关注的点：

- 教师课表接口存在受控业务异常，需要从后端数据或接口逻辑层面修复。
- 当前性能测试为基础可用性验证，不等同于完整压测，不能证明系统在高并发下表现稳定。
- 安全测试为基线扫描，不覆盖完整渗透测试，例如越权写入、文件上传、CSRF、复杂 SQL 注入等。
- 自动化测试默认不执行删除、审核、充值、导入等会修改真实数据的操作，因此对写操作覆盖有限。
- 前端页面测试主要验证路由和应用壳可访问，不替代完整 UI 交互测试。
- 如果测试账号权限、菜单或业务数据发生变化，部分登录态用例结果可能随环境变化。

## 10. 改进方向

### 10.1 后端改进

- 修复 `/edu/timetable/my` 在教师课表数据缺失时返回业务 500 的问题。
- 所有只读列表接口应统一处理空数据，返回空数组或空分页对象。
- 将常见业务异常从“系统繁忙”细化为更明确的错误信息，便于定位。
- 增加后端接口单元测试和集成测试，覆盖学生、教师、管理员三类角色。

### 10.2 前端改进

- 对无数据状态进行明确展示，例如“暂无课表”“暂无消息”“暂无审批记录”。
- 在接口返回受控错误时展示友好提示，而不是空白页或通用错误。
- 增加关键页面的端到端交互测试，例如登录、查看课表、提交报修、查看消息。

### 10.3 测试体系改进

- 增加写操作测试的隔离环境，使用专门测试库或自动回滚策略。
- 增加 Playwright 类浏览器自动化测试，覆盖真实点击、表单输入和页面跳转。
- 增加持续集成流程，每次后端或前端部署后自动执行冒烟测试和安全测试。
- 增加更完整的性能测试，包括并发用户数、吞吐量、P95/P99 响应时间和资源使用率。
- 将 AI 总结和测试报告归档，形成历史趋势，观察通过率、耗时和失败类型变化。

## 11. 验收结论

基于本次 `127` 条自动化用例执行结果，Campus Platform 当前部署版本整体达到可验收状态。全量测试最终 `126` 条通过、`1` 条跳过，跳过项为教师课表接口的受控业务异常，不影响测试框架运行，也未发现大面积功能不可用、安全敏感路径暴露或部署配置错误。

建议将本次测试报告作为阶段性交付材料，同时将教师课表接口作为后续修复项跟踪。修复完成后，可优先执行接口测试和全量测试进行回归验证，确保跳过项恢复为通过。
