# 一体化智慧校园系统测试方案

## 一、项目现状摸底

本项目是一套一体化智慧校园系统，整体分为前端和后端两个工程：

- 后端工程：`campus-platform-backend-main`
- 前端工程：`campus-platform-frontend-main`

后端采用 Java 17 + Spring Boot 3.2.4 + Maven 多模块结构，主要包含 `campus-common` 公共模块和 `campus-server` 业务服务模块。系统依赖 MySQL 8.x 和 Redis 7.x，认证鉴权使用 Sa-Token，数据访问使用 MyBatis-Plus，接口文档使用 Knife4j。

前端采用 Vue 3 + Vite + Element Plus + Pinia + Vue Router，主要负责登录、首页看板、系统管理、教研管理和后勤服务等页面展示与操作。

从代码和文档来看，系统已经覆盖以下核心业务域：

1. 认证模块：验证码、注册、登录、退出、当前用户信息。
2. 系统管理：用户、角色、菜单、字典、操作日志、登录日志。
3. 教研管理：课程、排课、课件资料、课程评价、考勤、请假、成绩、成绩申诉。
4. 后勤服务：公告、宿舍、报修、校园卡、图书借阅。
5. 数据看板：首页统计、业务概览、快照刷新。

本次测试需要重点关注系统的权限隔离、业务状态流转、数据一致性、文件上传安全、Redis 缓存行为和前后端联调稳定性。

## 二、测试目标

本轮测试目标是验证智慧校园系统在本地开发环境下能够稳定运行，并保证核心功能流程、权限控制和异常处理符合预期。

具体目标如下：

1. 验证前端页面可以正常访问，页面路由、菜单展示和接口调用流程正常。
2. 验证后端接口在正常参数、异常参数、无权限访问等场景下返回结果正确。
3. 验证管理员、教师、学生三类典型角色的权限边界清晰。
4. 验证 MySQL 数据写入、更新、删除、分页查询和关联查询结果正确。
5. 验证 Redis 相关能力可用，包括验证码缓存、签到码缓存、看板缓存等。
6. 验证关键业务流程闭环，例如登录、公告发布与已读、课程排课、签到、请假审批、成绩归档与申诉、报修流转、图书借还。
7. 发现系统当前测试覆盖不足、自动化测试缺口和上线前风险项。

### 3.2 前端页面测试范围

前端地址为 `http://localhost:5173/`，重点测试以下页面：

1. 登录页：验证码展示、登录提交、错误提示、跳转逻辑。
2. 首页看板：统计卡片、图表展示、接口异常提示。
3. 系统管理：用户、角色、菜单、字典、日志页面。
4. 教研管理：课程、课表、考勤、请假、成绩、评价页面。
5. 后勤服务：公告、宿舍、报修、校园卡、图书页面。
6. 异常页面：404 页面、无权限访问提示、接口失败提示。

### 3.3 不在本轮重点测试的内容

本轮不重点覆盖以下内容：

1. 大规模性能压测。
2. 第三方系统真实对接。
3. 生产环境部署验证。
4. 移动端专项适配。
5. 安全渗透测试。

这些内容可以作为后续专项测试单独安排。

## 四、测试环境

### 4.1 本地环境要求

| 环境项 | 要求 |
|---|---|
| 操作系统 | Windows 10/11 |
| JDK | JDK 17 |
| Maven | Maven 3.8+ |
| Node.js | 20.19+ 或 22.12+ |
| npm | 10+ |
| 数据库 | MySQL 8.x |

| 浏览器 | Chrome / Edge |

### 4.2 服务地址

| 服务 | 地址 |
|---|---|
| 前端开发服务 | `http://localhost:5173/` |
| 后端接口服务 | `http://localhost:8080/api` |
| Knife4j 文档 | `http://localhost:8080/api/doc.html` |
| MySQL | `127.0.0.1:3306` |
| Redis | `127.0.0.1:6379` |

### 4.3 启动步骤

后端启动：

```bash
cd campus-platform-backend-main
mvn install -DskipTests
cd campus-server
mvn org.springframework.boot:spring-boot-maven-plugin:3.2.4:run -DskipTests
```

前端启动：

```bash
cd campus-platform-frontend-main
npm install
npm run dev
```

数据库初始化：

```bash
mysql -u root -p < sql/campus_system_campus.sql
```

如果拆分执行 SQL，也需要按系统表、教研表、校园服务表的依赖顺序导入。

## 五、测试数据准备

测试前需要准备管理员、教师、学生三类账号，并保证角色权限和菜单权限配置正确。

| 角色 | 建议账号 | 用途 |
|---|---|---|
| 管理员 | `admin` | 系统管理、基础数据维护、全量接口权限验证 |
| 教师 | `teacher01` | 课程、排课、考勤、请假审批、成绩录入 |
| 学生 | `student01` | 查看课表、签到、请假、查看成绩、报修、借书 |

基础数据建议包括：

1. 至少 2 个班级，例如计算机 2401 班、计算机 2402 班。
2. 至少 2 门课程，例如高等数学、数据结构。
3. 至少 1 名教师绑定课程。
4. 至少 2 名学生绑定班级。
5. 至少 1 栋宿舍楼、2 个宿舍房间。
6. 至少 3 本图书，覆盖可借、库存为 0、已借出场景。
7. 至少 2 条公告，覆盖全员公告、角色公告、班级公告。

## 六、测试策略

### 6.1 单元测试

后端已有部分 JUnit5 + Mockito 测试，主要集中在 `svc` 模块控制器层。单元测试重点关注：

1. 参数校验是否正确。
2. 服务层调用是否符合预期。
3. 权限或当前用户判断是否正确。
4. 状态流转非法分支是否被拦截。
5. 异常场景是否返回明确错误。

建议执行：

```bash
cd campus-platform-backend-main/campus-server
mvn test
```

### 6.2 接口测试

接口测试可以使用 Apifox、Postman 或 curl。测试时需要先登录获取 token，再把 token 放到请求头：

```http
Authorization: <token>
```

接口测试重点关注：

1. 正常请求返回 `code=200`。
2. 参数缺失、类型错误、非法状态返回明确错误。
3. 未登录访问返回 401。
4. 无权限访问返回 403 或业务错误。
5. 业务异常返回信息清晰。

### 6.3 前端联调测试

前端联调以手工测试为主，重点验证：

1. 页面能否正常加载。
2. 菜单和路由是否按角色展示。
3. 表格分页、搜索、筛选是否正常。
4. 新增、编辑、删除弹窗交互是否正常。
5. 操作成功和失败提示是否清晰。
6. 登录失效后是否能回到登录页。

### 6.4 集成测试

集成测试重点验证后端与 MySQL、Redis 的真实交互：

1. 数据写入后能正确查询。
2. 事务失败时不会产生脏数据。
3. Redis 缓存 TTL 符合业务设置。
4. 并发扣减库存、床位、签到等场景不会出现数据不一致。
5. 文件上传后数据库记录和物理文件保持一致。

### 6.5 回归测试

每次修复缺陷或修改核心模块后，需要回归以下基础链路：

1. 登录获取 token。
2. 管理员新增用户并分配角色。
3. 教师创建签到，学生完成签到。
4. 学生提交请假，教师审批。
5. 教师录入成绩，管理员审核归档，学生提交申诉。
6. 学生提交报修，管理员受理和完成，学生验收。
7. 学生借书和还书，库存正确变化。

## 八、异常和边界测试

本轮需要补充以下异常和边界场景：

1. 空字符串、超长字符串、非法数字、负数、非法枚举值。
2. 不存在的 ID，例如不存在的用户、课程、图书、宿舍房间。
3. 重复提交，例如重复登录、重复签到、重复借书、重复挂失。
4. 越权访问，例如学生修改他人成绩、学生验收他人报修单。
5. 文件上传异常，例如空文件、超大小文件、非法后缀文件、重复文件。
6. 分页边界，例如 pageNum 为 0、pageSize 过大、无数据页。
7. 缓存边界，例如验证码过期、签到码过期、Redis 不可用。
8. 并发边界，例如多人同时借同一本库存很少的图书、多人同时分配最后一个床位。

满足以下条件可认为本轮测试通过：

1. 前端和后端服务可以稳定启动。
2. 核心接口冒烟测试全部通过。

## 十一、缺陷等级定义

| 等级 | 说明 | 示例 |
|---|---|---|
| P0 | 系统无法启动或核心链路完全不可用 | 后端启动失败、登录接口不可用 |
| P1 | 核心业务不可用或存在严重数据错误 | 成绩归档后仍可修改、借书库存扣减错误 |
| P2 | 普通功能异常但有替代路径 | 某个筛选条件失效、导出格式异常 |
| P3 | 展示、文案、交互体验问题 | 提示语不清晰、按钮间距异常 |

## 十二、当前风险和缺口

1. 后端自动化测试覆盖面不够，目前已有测试主要集中在 `svc` 控制器层，`auth`、`sys`、`edu` 还需要继续补充。
2. 缺少 MySQL 和 Redis 的集成测试，缓存 TTL、事务一致性、并发扣减等风险需要靠手工或专项测试发现。
3. 前端当前没有配置 Vitest、Cypress 或 Playwright，页面回归主要依赖手工测试，效率和稳定性有限。
4. 文件上传、Excel 导入导出、权限菜单刷新这类跨前后端功能容易出现联调问题，需要重点回归。
5. 本地 Maven 默认 Java 版本可能不是 JDK 17，启动后端时需要确认 `JAVA_HOME` 指向 JDK 17。

## 十三、后续改进建议

1. 给 `auth`、`sys`、`edu` 模块补充控制器和服务层单元测试。
2. 引入集成测试，覆盖 MySQL、Redis、Mapper、事务和缓存 TTL。
3. 前端引入 Vitest 做组件和工具函数测试。
4. 使用 Playwright 或 Cypress 覆盖登录、权限菜单、核心业务流程。
5. 把接口测试集合沉淀到 Apifox 或 Postman，形成可重复执行的回归集。
6. 在 CI 中加入后端 `mvn test` 和前端 `npm run build`，避免基础构建问题进入主分支。

## 十四、结论

本测试方案覆盖了一体化智慧校园系统当前主要的前后端功能、接口模块、角色权限、业务状态流转和数据一致性场景。按照本方案执行后，可以较系统地验证项目在本地开发环境下的可用性和核心质量。

当前项目已经具备基本的业务功能和部分后端单元测试，但自动化测试、集成测试和前端端到端测试仍然不足。建议先以本方案中的冒烟测试和核心业务流程作为近期验收标准，再逐步补齐自动化测试体系。
