# 技术规格文档

## 1. 文档概述
本文档是基于《一体化智慧校园系统需求说明书》与《需求分析文档》编写的技术规格说明书。旨在从实现视角，详细阐述系统的总体架构、各模块技术方案、数据库及接口规范等，为开发团队提供明确的技术依据与实现指南。

## 2. 总体技术架构设计
本系统采用前后端分离架构，核心基于 Spring Boot + Vue3 构建。
- 前端架构：以 Vue3 为核心框架，使用 Vite 进行极速构建。通过 Pinia 进行全局状态管理（如用户信息、Token、动态路由状态），VueRouter 控制前端页面路由流转。HTTP 请求使用 Axios 封装统一处理，UI 层采用 Element Plus 组件库。
- 后端架构：采用 Spring Boot 3.x（基于 Jakarta EE 规范）作为核心基础框架，分为 Controller（控制层）、Service（业务逻辑层）、Mapper（数据访问层）。数据持久层集成 MyBatis-Plus 简化 CRUD 操作；数据库连接池采用 Spring Boot 3.x 默认内置的 HikariCP；安全与权限控制集成 Sa-Token 框架；缓存中间件采用 Redis 7.x，应用层缓存采用 JetCache（本地 Caffeine + 远端 Redis 双层）加速热点数据访问；通用工具包采用 Hutool（BCrypt 密码散列、验证码、Excel解析等）；文件存储采用本地磁盘归档（生产环境可平滑替换为 MinIO/OSS）。
- 存储与数据库：核心数据持久化采用 MySQL 8.0.32，规范化支持逻辑删除（is_deleted）及标准的审计时间字段（create_time、update_time）。

## 3. 系统模块技术设计

### 3.1 登录与权限模块
- 前端页面：登录主页包含学号/工号/账号输入框、密码输入框及简化版图形验证码组件。
- 后端分层：Controller 层接收登录请求并校验图形验证码；Service 层负责调用 Hutool BCrypt 比对密码散列值及基于 Sa-Token 执行签发 Token。
- 关键接口：POST /auth/login 用于账号密码登录验证并下发 Token；GET /auth/userInfo 用于获取当前登录人基础信息与权限标识。
- 权限控制：基于 Sa-Token 实现 API 级鉴权，所有非放行接口均需校验 Token。前端根据权限拉取动态菜单。
- 缓存策略：Token 状态及过期期（默认2小时）由 Sa-Token 底层管理；图形验证码的存活期（5分钟）交由 JetCache 存储。
- 日志点：在登录接口的 AOP 切面中，记录状态、IP 至 sys_login_log。
- 异常处理：连续3次密码错误抛出特定业务异常，触发缓存计数器锁定该账号1小时拦截请求（管理员可手动解锁）。

### 3.2 系统管理模块（仅管理员）
- 前端页面：用户管理台账基于 Element Plus 的 Table、Pagination 组件，附带 Excel 导入按钮；同时提供数据字典维护模态框、角色权限分配树组件。
- 后端分层：Service 层负责复杂业务，如通过 Hutool 解析导入用户并校验唯一性。Mapper 层执行分页查询。
- 关键接口：POST /sys/user/import 接收文件解析 Excel 批量导入用户；PUT /sys/dict 更新或新增数据字典。
- 权限控制：接口硬性添加防越权注解，拦截越权访问。
- 缓存策略：对于新增或修改的数据字典，在写入操作完成后，显式调用 JetCache 的缓存清理或更新注解，保持数据一致性。
- 日志点：使用自定义注解标记重大写操作（增删改），记录操作人、类型与入参至 sys_operate_log。
- 异常处理：捕获 Excel 导入中的缺失必填项或数据库主键冲突异常，向前端抛出具体的行号错误信息并回滚事务。

### 3.3 课程管理模块
- 前端页面：课程检索列表页、课件资料上传弹窗（整合 Upload 组件限制容量及格式）、多视图课表渲染组件、星级与文本结合的课程评价表单。
- 后端分层：Service 负责串联教师、班级、课程之间的业务规则约束关系；Controller 层解析文件上传流。
- 关键接口：POST /edu/course/material/upload 提供教师课件资料上传；GET /edu/course/timetable 获取学生或教师的课表聚合数据。
- 权限控制：资料上传接口限定仅教师角色可访问；评价接口参数需强校验当前 Token 对应的学生是否在该课堂的名单中。
- 缓存策略：对于查询频率极高的课程简介等基础信息设定 JetCache 缓存，过期策略设定为12小时。
- 日志点：管理员分配或变更课程执教教师时，记录变更轨迹日志。
- 异常处理：文件上传业务拦截器校验 MIME 类型，若非文档、图片或体积超过10MB，抛出文件受限异常。

### 3.4 考勤管理模块
- 前端页面：教师端考勤任务下发配置板；学生端的一键签到悬浮区及请假申请表单（包含提交事由、起止时间）；考勤统计宏观报表页。
- 后端分层：考勤场次创建时，后端生成场次 ID 及其倒计时状态；学生签到请求到达 Service 层时，核心比对当前时间戳与该场次有效时效。
- 关键接口：POST /edu/attendance/start 供教师发起并在后台开启签到场次；POST /edu/attendance/sign 供学生发起签到响应。
- 权限控制：学生签到请求需在 Service 层严密校验当前账号是否属于该被考勤的课程与班级名单内。
- 缓存策略：由于签到动作存在瞬时高并发可能，考勤场次有效性通过短效缓存标识进行极速核验，签到成功记录亦可暂入缓存再异步处理落库。
- 日志点：教师若对学生某次缺勤记录进行手工补签或系统状态修改，必须在相关操作中留下操作痕迹溯源。
- 异常处理：利用场次ID与学生ID的组合作为分布式防重键，捕获并忽略因网络抖动引起的重复签到提交事务。

### 3.5 成绩管理模块
- 前端页面：教师专属分数导入与小窗补录页；管理员终审拦截确认与备注页；学生端带有图片上传佐证功能的复议申诉对话框。
- 后端分层：基于有限状态机标识进行流转，包括录入待审阶段、审核通过归档阶段、驳回修改阶段。
- 关键接口：POST /edu/score/batch-import 按模板批量解析分数值并存放至待审区；PUT /edu/score/audit 管理员裁定审核通过或带有理由的驳回。
- 权限控制：已进入待审核及以后状态的数据列，即刻锁定对应教师的修改入口权限；学生个人仅能查询自身匹配的成绩，校内排行屏蔽具体分数。
- 缓存策略：已终极归档的往期完整学期成绩清单在后台计算好相关统计量后整体放入缓存，避免高频度读取引发数据库聚合函数的重复大算力消耗。
- 日志点：每一次管理员执行驳回退回动作，必须关联记录具体的退回文书意见与审核操作时间节点备查。
- 异常处理：后端校验拦截超出百分制正常域范围或非常规字符的分数写入操作，抛出异常并携带确切的报错坐标行列供前端修正。

### 3.6 宿舍、报修及校园服务模块
- 前端页面：图文组合附加紧急度标识的后勤报修申请表单；携带处理状态时间轴的进度跟踪面板；校园卡消费简单账单流列表页。
- 后端分层：报修工单处理采用状态机制映射不同的维修推进阶段；校园卡与图书借阅对接校内外接口则应在 Service 层封装 Http 调用动作。
- 关键接口：POST /campus/repair/apply 提供多图上传的报损单投递；GET /campus/card/records 按自定义时间范围提取消费记录结果流集。
- 权限控制：对于核心房源床位分配写入接口严加校验管理员权限；学生只允许查阅自身及其所在同寝室人员的基础分配信息，以及己方实名消费流水。
- 缓存策略：针对高频查询如图书基础目录、宿舍楼栋字典等提供常规缓存支持；而个人消费账务流水及报修进度要求绝对实时对接到持久层数据探查。
- 日志点：报单从初始申请到维修工接棒处理中再到最终完结收口的每一个阶段跳变操作轨迹严格留痕记录。
- 异常处理：预约图书等扣减动作执行前采取防超卖的并发管控机制，对存量等于零的请求果断阻断抛出库存耗尽的异常反馈。

### 3.7 数据统计与公告模块
- 前端页面：利用折线图与柱状图展示全校考勤率与后勤报修完结率的管理侧大盘；集成常规富文本编辑器的公告发布台及前端头部的未读红点站内信信箱。
- 后端分层：大盘综合数据抓取高度依赖 Service 侧聚合多表关联的产出视图；系统级别公告向预定群组标签发布完毕后，异步切入驱动对应人员的消息计数器累增。
- 关键接口：GET /dashboard/admin 获取提供给高层视野的全套核心统计数值卡片；POST /campus/notice 发布公告及下发定向受众规则设配。
- 权限控制：数据控制台严格按照管理层、教师工作台、学生个人看板划分了三级隔离数据源，严禁下层越级查看不合身份级别的宏观汇总。
- 缓存策略：针对耗时的统计聚合大屏数据，采用特定的定时任务调度并按设定周期生成快照推送进入缓存区，大幅度削减单次请求带来的复杂查询拖累。
- 日志点：管理层发布重要公共通告的操作节点与下发覆盖波及的群众面积等相关行为记录入档核查。
- 异常处理：前端查询大运算图表接口面对突发超时响应时应主动进行降级平替机制（如展示上一轮安全快照），保护后端免受系统级查询拖累死机险情。

## 4. 数据库技术规格
- 规范字段：系统中所有核心业务表必须包含主键自增 id，且均硬性要求预留 create_time 创建时间、update_time 更新时间，以及作为逻辑删除管控标志的 is_deleted 字段（0为正常、1为已删除）。
- 关联约束：完全阻绝物理级别的外键强连结关系部署。各类业务级数据关联的完整性维护必须后置在 Spring Boot 应用层的 Service 中，通过事务一致性和严格的空值探测逻辑进行接管达成。
- 命名法则：必须遵守规范的下划线分词定义法则（例如 course_code）。针对各板块领域依据自身职能归属必须挂接统一的表名前缀（主要划分为 sys_系统表、edu_教学表、campus_校园辅助表）。
- 索引策略：针对登入作为身份依据的学号或工号，以及高度频密被检索的复杂标识词（如单篇核心课程编号等），必须进行唯一性索引创建以强化提取比中检索效能和强制反重防线构建。

## 5. 接口设计规范
基于 RESTful 风格指引，所有数据通讯接口端实施 JSON 强制返回约束（注：为避免代码块格式，此处以文本说明组织结构）：
统一交互响应体外壳套件包含三组标量 code、msg、data。
- code 指定 200 为动作顺达通行、400 为传输参量检测防查拒载、401 明示 Token 失效剥离及身份流失、403 表示无权限访问挡关、500 指向后端不可预断的服务器异常。
- 遭遇反馈目标呈现空数据时，data 部分必须规整吐出 null 绝不可出现整体键值层脱位的作法。
所有带有分页查阅展现性质的集合接口，应当标准化返回四组基础支撑导航参量段：total （底量计件总规模）、list （当前页数据列表）、pageNum （当前页码）、pageSize （每页条数）。

## 6. 安全设计
- 密码加固：严格杤绝明码文的存储。任何进入入池建档的用户原始密码均必须被通过 Hutool 提供的 BCrypt 算法进行不可逆单向散列处理后方可入库存储，登录时通过 BCrypt.checkpw() 比对散列值而非解密，杤绝后台视查直取。
- 无害清洗：搭配引入全局过滤网罩专门除脏洗脱随前端参数捎带送入夹杂隐埋的注入脚本和危险串（XSS防范）；ORM 层面利用 MyBatis-Plus 预编译特性消解 SQL 注入。
- 越权拦截：搭配引入全局 Sa-Token 权限体系拦截越权访问。深入 Service 层私主数据调用（诸如本人成绩查访、个人信报查阅核收）等操作时，不仅要校验接口权限，还必须在逻辑中比对当前访问实体账户与数据归属是否同源契合。

## 7. 缓存设计
- 缓存主架构选定：系统选择部署并依靠配置侵入性低、性能突出的 JetCache 加挂支持体系。
- 动态隔离分期策略：
针对变动频率低的热读取数据（如基础配置字典、菜单体系大集），配置常线存储期（12小时），并给修改触发器配属联动缓存刷新清理策略避免长时读错的假象呈现。
对于核心鉴权 Token 令等高敏属性实行内部自驱管控，设定生命周期（2小时活动有效期）管理。

## 8. 前端技术规格
- 页面与组件搭构模式：整体搭建在纯 Vue3 提供出的组合式 API 环境体系里。遇大厚重沉余业务区应按细微粒度抽出通用剥离小组件复合并用。
- 中央状态控制：利用优效管理机能管家系统 Pinia ，中心化纳管揽存用户信息、权限树等全局履历凭证池子，以代替凌乱纷飞的散式属性父子层层传递机制方案。
- 统一请求中枢：网络层封包交汇均集归并拢托放进 Axios 请求网全盘管控。通过配置代理拦截体系进行统一鉴权头封装、响应过滤和 Token 失效自动重定向回登录页的规范运转。
- 对齐适屏张放：大结构与框栏设计依托 Element Plus 响应式栅格与 Flex 弹性布局构建，使得展示大区在各种标准带屏幕大小（从1366到1920分辨率宽度）里均兼容匹配不死乱不变形展图完备。

## 9. 后端技术规格
- 脱耦断连专职分配规章：坚持贯彻到底三层解耦的隔墙断层切割分解。限定 Controller 端只办客从迎送的报文验参解析不参与具体脑算精算；把权限业务联动控制决判放止执行权归由专属 Service 单独吞食；合理利用 Hutool 中预制优效简化的通用短工具取代各种冗长耗时基础代码从而激发出开发降本增效的极致产能释放。
- 异常统一防漏控系统：建立起基于 Spring 顶端的全局异常一统化收拢布网拦截体系，将层出不同五花各色乱状底层报错悉数吞下捕获并统一转换规范为标准化 JSON 语言模块回送前端展示。强行压降断绝将带有服务器详细堆栈绝对路径全盘回呈大露白直呈给终端用户所引起引发的巨大安全漏洞危机的发生概率。

## 10. 部署与运行建议
- 环境硬性准接方案：所配置发车运行服务端机房应装备配置 JDK 17 及之后版本的稳定主推版底盘，持久化架构关联配置 MySQL 8 基石为核载供存储服务。
- 投线装列产件制式：前端靠执行 Vite 命令构建出轻量化静态前端资源包文件用以向后供 Nginx 挂载。后位服务端主躯则统合规制打包成包含一切依赖件的胖包（Fat-Jar）结构，依赖标准化配置文件快速启动提供后端接口支撑实际业务运转效力。

## 11. 技术风险与规避方案
- 过峰流量穿透挤宕隐患之应对：对于类似于放课集聚时点下的大股巨流量瞬时突击触发（例如全堂抢报签到考勤、查分抢榜），这种急流极易引发计算层紧绷死盘及连接池告急。主应对方案为前置短效轻缓存盾阵挡住其查询锐流不使直接频繁打向数据库；另外同步关注调整连接池（HikariCP）监控参数进行闲闭忙张调节缓冲并发压力。
- 脏数据死僵隐危阻控防治策略：本系统彻底推翻抛弃了数据库里采用物理外键关联强拉勾引绳索的方案，换取了高度灵活用法。但同时也极易诱发斩除源数据却漏抛子连数据的垃圾残留。此症的根本解决规避点就是要在相关业务断层逻辑入口即 Backend Service 抹除逻辑口上加高装牢检测防逃验卡机制，感知有依赖牵连未平结残留时强制返回拦截，要求先决结清子账方可再斩主刀销毁主记录。
