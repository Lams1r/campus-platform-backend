# Swagger 中文化与 API 乱码修复设计

## 背景

当前项目使用 Knife4j + OpenAPI 3 生成接口文档，但 Swagger 页面中的项目标题、接口名称、参数说明和模型字段说明普遍缺失中文语义，前端联调时需要反复对照源码理解接口含义。

同时，项目内已有一批 API 相关字符串和注释出现乱码，影响 Swagger 页面展示，也影响校验提示和统一返回消息的可读性。

## 目标

1. 让 Swagger/Knife4j 页面中的项目标题、模块分组、接口摘要、参数说明、请求体和响应体字段说明显示为中文。
2. 修复 API 对外可见层中的现有乱码，包括 Swagger 配置文案、校验提示、统一返回说明和接口注释相关文本。
3. 不修改现有接口路径、字段名或返回结构，避免影响前后端联调协议。

## 范围

本次改动限定在 API 对外可见层：

- OpenAPI 配置：项目标题、描述、模块分组。
- Controller 层：控制器标签、接口摘要、常见参数说明。
- 模型层：DTO、VO、公共返回包装，以及直接暴露给 Swagger 的实体类字段说明。
- API 可见乱码：校验提示、返回消息、Swagger 配置文案。

不包含：

- 全仓库所有中文注释的全面清洗。
- 接口字段重命名。
- DTO 重构或接口协议调整。

## 方案比较

### 方案一：仅修 Swagger 配置和少量接口标题

只修 OpenAPI 标题、描述和个别控制器注解。

优点：
- 改动最小。

缺点：
- 参数与模型字段依旧以英文名为主，前端使用价值有限。

### 方案二：完整 API 中文化

为控制器、接口、参数、DTO/VO、公共返回结构和直接暴露的实体补充 OpenAPI 中文注解，同时修复 API 可见乱码。

优点：
- 能直接解决前端联调可读性问题。
- 改动集中在文档层和可见文案层，风险可控。

缺点：
- 需要逐个接口面补充说明，工作量中等。

### 方案三：全仓库乱码治理

在方案二基础上修复全部模块中的乱码。

优点：
- 最彻底。

缺点：
- 范围明显扩大，容易偏离当前诉求。

## 选型

采用方案二。

理由：
- 直接服务 Swagger 前端联调场景。
- 不把任务升级成全仓库文案治理。
- 能在不改接口协议的前提下显著提升接口文档质量。

## 设计

### 1. OpenAPI 配置层

- 修正 `campus-server/src/main/java/com/campus/system/config/SwaggerConfig.java` 中的标题、描述和联系人说明。
- 按模块补充分组配置，使 `auth`、`sys`、`edu`、`svc` 在文档中按业务域展示。

### 2. 接口展示层

- 为控制器补 `@Tag(name = "...")`。
- 为公开接口补 `@Operation(summary = "...")`。
- 为路径参数、查询参数、上传参数补 `@Parameter(description = "...")`。
- 对必要的接口补充最少量 `@ApiResponse`，重点说明成功响应，不做过度装饰。

### 3. 模型展示层

- 为 DTO、VO、统一返回结构补 `@Schema(description = "...")`。
- 为直接暴露到 Swagger 的实体类补类和字段级说明。
- 为分页结构说明 `total`、`list`、`pageNum`、`pageSize` 等字段语义。

### 4. 乱码修复策略

- 只修 API 可见层中的乱码字符串。
- 对原本已经被错误保存成乱码的字符串，直接替换成正确中文文案。
- 对仅用于内部实现的普通注释，不做无边界清理。

## 风险与约束

- 不修改接口路径、字段名、状态码或返回结构。
- 如果实体直接作为请求体或响应体暴露，优先补充文档注解，不扩展成 DTO 重构。
- OpenAPI 测试优先验证关键中文文案是否出现在文档输出中，避免只依赖人工判断。

## 验证

### 自动验证

- 新增 OpenAPI 文档测试，验证 `/v3/api-docs` 中存在关键中文标题、接口摘要和字段描述。
- 运行受影响模块测试与编译，确保注解和字符串修复不破坏现有逻辑。

### 人工验证

- 打开 `doc.html` 检查项目标题、模块标签、接口摘要、参数说明和模型字段说明是否为中文。
- 抽查 `auth`、`sys`、`edu`、`svc` 各 1 到 2 个接口，确认覆盖不是局部性的。
