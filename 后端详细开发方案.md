# 一体化智慧校园系统 - 后端详细开发落地方案

## 1. 方案概述
本方案为《技术规格文档》中后端部分的深度延展与具体实施蓝图。旨在明确“一体化智慧校园系统”后端服务的项目结构、核心技术栈应用规范、通用核心机制实现细节及核心业务线的具体开发步调。本文档直接指导后端开发人员的编码工作，确保最终交付高质量、高性能的 RESTful API 服务。

## 2. 后端核心技术栈与版本基准
- **开发语言环境**：Java 17 (JDK 17)
- **核心框架引擎**：Spring Boot 3.x (基于 Jakarta EE 规范)
- **数据持久化中间件**：
  - MySQL 8.0.32 (InnoDB引擎，UTF8MB4字符集)
  - MyBatis-Plus (简化单表操作，提供通用 Wrapper 及分页组件)
- **数据库连接池**：HikariCP (Spring Boot 3.x 默认内置，高性能轻量级连接池)
- **安全与权限框架**：Sa-Token (轻量级，提供登录、多账号体系、角色权限拦截、踢人下线)
- **缓存中间件**：Redis 7.x (作为 JetCache 远端缓存存储、Sa-Token 分布式会话存储基座)
- **多级缓存加速框架**：JetCache (提供本地 Caffeine + 远端 Redis 双层缓存支持，支持方法级注解缓存)
- **文件存储方案**：本地磁盘存储 (开发期，文件统一归档至项目配置的 `upload-path` 目录，生产环境可平滑替换为 MinIO/OSS)
- **综合工具包组件**：Hutool (使用其提供的 BCrypt 密码散列、图形验证码生成、Excel 导入导出验证等子模块)

## 3. 项目工程结构规范 (Package 划分)
项目采用主流的按层与按模块混合切割的 MVC 结构，根包命名规范为 `com.campus.system`，子包结构设计如下：
```text
com.campus.system
├── config        // 系统全局配置 (MyBatisPlusConfig, WebMvcConfig, SaTokenConfig, JetCacheConfig等)
├── common        // 存放系统的公共元素
│   ├── api       // 统一接口响应体 (Result<T>)、分页包装类 (PageResult<T>)
│   ├── enums     // 业务枚举字典 (如角色类型枚举、审批状态枚举)
│   ├── exception // 全局自定义业务异常 (BusinessException) 与全局异常捕捉处理器 (GlobalExceptionHandler)
│   └── constants // 常量类 (如缓存前缀名、系统常量)
├── filter        // Servlet 或框架拦截器 / 基础 Filter (如基于 Sa-Token 的前置路由拦截、MDC日志追踪)
├── annotation    // 自定义注解体系 (如 @LogRecord 切面操作日志注解)
├── util          // 二次封装的工具类，如 Sa-Token获取当前用户ID工具、OSS上传工具
└── modules       // 业务领域聚合块 (按大模块划分)
    ├── auth      // 登录授权、登出、验证码等核心认证入口
    ├── sys       // 系统底座 (用户管理、字典管理、操作日志)
    ├── edu       // 教学考评 (课程、考勤签到、成绩录入申诉)
    └── svc       // 校园服务 (后勤报修、一卡通流水接口、看板大屏视图)
        ├── controller // 接口暴露层 (必须做参数基本校验)
        ├── service    // 业务逻辑层接口 (存放复杂流转策略)
        │   └── impl   // 业务逻辑实现类 (挂载事务 @Transactional)
        ├── mapper     // MyBatis 映射层 (继承 BaseMapper<T>)
        ├── entity     // 数据库实体映射类 (@TableName, @TableId 等注解)
        └── dto        // 跨层交互传输对象 (RequestDTO 接收请求, ResponseVO 响应视图)
```

## 4. 全局核心机制的落地实现
为杜绝各开发人员各行其是，本方案圈定以下全局底盘机制的代码开发准则：

### 4.1 统一的 RESTful 响应报文规范
任何一个 Controller 返回的数据，强制使用 `Result<T>` 进行包装，不准许直接抛出裸数据实体：
```java
public class Result<T> {
    private Integer code;    // 状态标识 (200:成功, 400:传参驳回, 401:Token断供, 403:权限不足, 500:服务器不可预知错误)
    private String msg;      // 友好的人机阅读提示语
    private T data;          // 核心净荷载体 (查不到带回 null)
}
```

### 4.2 统一的异常全局拦截网兜
基于 `@RestControllerAdvice` 编写 `GlobalExceptionHandler`，职责包含：
1. 精准捕获业务抛出的 `BusinessException`，将封装的 code 与 msg 平铺为 `Result` 回显。
2. 捕获 Sa-Token 抛出的 `NotLoginException` / `NotPermissionException`，转化为 `401`/`403` 标准回信。
3. 捕获并拆解后端 `@Validated` 层传导出的参数非法异常（`MethodArgumentNotValidException`），提取第一条错漏信息归并为 `400` 打回前端。
4. **最后一道防线**：用 `Exception.class` 捉住一切未捕获的系统崩溃错误，打印控制台 Error 日志，但给前端统一掩盖反馈报修代码如“系统繁忙请求异常”，严防堆栈路径红字暴露。

### 4.3 底层数据字段的强制共识
- 所有实体基类继承一个底盘基盘：
```java
public abstract class BaseEntity {
    @TableId(type = IdType.AUTO) // 强制自增主键
    private Long id;
    
    @TableField(fill = FieldFill.INSERT) // Insert时组件自动回填切面
    private LocalDateTime createTime;
    
    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;
    
    @TableLogic // 宣告这是一个供 MyBatis-Plus 使用逻辑防除防线的标记
    private Integer isDeleted;
}
```
- 外键强制抹杀：所有跨实体的约束均在 Service 层利用 `Long XXXId` 作桥接，查询校验合法性后操作，斩断物理层面的硬连结。

## 5. 三大防线安全机制布网
### 5.1 RBAC + Sa-Token 路由与 API 拦截防权限穿透体系
1. **鉴权下发**：用户验证账户通过后，使用 `StpUtil.login(userId)` 系统级植入凭证；使用 `StpUtil.getTokenValue()` 封装进 `Result` 交给前端。
2. **路由拦截**：配置 `SaTokenConfig` 注册全局拦截器，放行白名单（`/auth/login`, `/auth/captcha`），拦截其余所有 `/**` 的探访进行 Token 有效期验真。
3. **接口细粒度注解把控**：系统管理面板的方法使用 `@SaCheckRole("admin")` 严守；特定教研功能使用 `@SaCheckPermission("course:add")` 精细化约束拦截越界入侵。

### 5.2 敏感动作加持日志防消痕追踪网
利用 AOP 机制开发 `@LogRecord(module = "用户管理", type = "新增")` 自定义注解。织入 Controller 切面，当该类改变数据池的操作接口完结或崩溃后，AOP 会抓取探针如：操作者ID(由 `StpUtil.getLoginId()` 拉取)、耗时毫秒、入参 JSON 化备份，直接单开后备异步线程 (`@Async`) 追加落盘入 `sys_operate_log` 中备查。

### 5.3 数据输入级防毒面罩
通过针对实体类接收参数的（`RequestDTO`）全方位加上 `jakarta.validation` 标准化注解（Spring Boot 3.x 已全面迁移至 Jakarta 命名空间）：`@NotBlank(message = "学号不可为空")`，`@Size(max=200)` 等约束，把具有破坏力或冗余脏的恶意入库企图，在 Controller 入城口处被全局异常处理网兜捕获遣返。

## 6. 重点业务模块后端攻坚指南

### 6.1 高通量场次下防拥挤考勤签到
- **场次预存缓存 (Cache-Aside)**：当教师点开签到任务，Service 首先生成具有生命倒计时的场次编码信息塞入 Redis/JetCache 之中（附带生命期30分钟）。
- **极速签入与防单机穿击**：当几百学生齐压签到键，校验不走 DB，直击 Cache 比对场次状态与是否在场有效期；验证放行后，结合考勤任务Id+个人Id合并做排斥主键拦截一人多点签到的违规操作；数据首先丢给内存消息层，后排安排调度程序缓冲刷血落入主库池。

### 6.2 断开篡改防线漏洞的分数复查流
- **阶段流转状态机驱动**：成绩表核心预置 `status` 枚举 (`0:脱机录入待审`, `1:高审被驳回需要重新修正`, `2:完美审批归档查封`)。
- **业务横切拦截（关键点）**：在所有承压改写动作（Update分数）进入持久层前，强加 Service 段查验：
  ```java
  if(scoreRecord.getStatus() == 2) {
      throw new BusinessException("终核分库已铸印查封，禁止进行改分僭越动作！");
  }
  ```

### 6.3 超耗能图表大屏提取与缓存分离机制
- **破除强聚聚合运算**：对于高阶领导要求调看统计（如全院签到曲线分布、后勤修复率大饼），绝对禁止客户端去击发 DB 去 `COUNT` / `SUM` / `GROUP BY` 千万级的明细记录表。
- **配置定时刷新作业库**：借由 Spring Boot 的 `@Scheduled(cron = "0 0/10 * * * *")` 激活定期的统算作业机器人，每隔例如10分钟，将复杂的跨表连缀宏观统计成果推导而出并存为一个完整版 JSON 快照扔进 JetCache ；而前端查询大屏面板接口的数据获取仅仅是做简便的去缓存桶里将快照拎出展示的传接动作。

## 7. 后端实施线路图 (Milestone 路线)
为迎合并跑整体的 8 周总工作计划，后端需遵循如下攻坚路段进度铺设：
- **实施周期 ①：底盘与权脉基建打通 (第1周~第2周)**
  - 交付：数据库物理建模成型；跑通 Spring Boot 框架全基础配置；完成用户增删改与验证码下行、账号登录比对、下发 Token 接口闭环打通；编写配置化全站防越权基座体系。
- **实施周期 ②：全要素教研闭环建设 (第3周~第4周)**
  - 交付：基于 MyBatis-Plus 挂载打通高频使用的教师辖带专属课程信息的查询及课件流上传接口；交付基于 JetCache 短持缓防并发的防拥打卡连线以及双跨师生的请假走单审验后端流程流转。
- **实施周期 ③：封核成绩盘与服务图大屏 (第5周~第6周)**
  - 交付：利用 Apache POI/Hutool 实现成绩块板从空表下装至数据重载批量吸入上传校验，写死防修改的权限判定口；编写定时器轮转服务抽取各种报修单/消费单的综合聚合金字塔数据至预存盘为大屏前端探取输出提供基站库口。
- **实施周期 ④：熔断防线抗压压测与完工交付 (第7周~第8周)**
  - 交付：配合 JMeter 跑大并发测试流对之前铺设的缓存机制验效阻断排查内存溢出点或连接池拥堵并修复配置调频比例；利用 Nginx + 自启脚本（或以 Docker 镜像形态）包将核心 Fat-Jar 后置服务器做完整闭环上线封盘。

## 8. 本章小结
本后端开发落地详案充当连串从顶层蓝图至底端落刀切肤实战的具体枢纽转化机能桥梁角色。把之前宣讲云端的规格设想死死咬实固化成具体如何配置 Maven、怎么用好 MyBatis-plus Wrapper，特别是将考勤防崩大坝、成绩防暗算锁死和数据拉取减荷避瘫等技术困局做了清晰直白的微操指导，全面保障随后即将打响的实弹编程战争中后端部队不散板不迷向方向精准高效发力。
