# 需求分析文档

## 1. 引言
### 1.1 编写目的
本文档旨在对“一体化智慧校园系统”进行全面细致的需求分析，明确系统的业务目标、功能边界与非功能属性。本文档将作为平台架构设计、数据库设计、前后端开发及最终测试验收的核心依据，为系统的研发提供明确的约束与指导，适用于专业课程设计或毕业设计文档的标准规范。

### 1.2 项目背景
随着教育信息化的发展，传统校园管理中存在数据孤岛、业务流程协同效率低等痛点。通过搭建一体化智慧校园系统，实现多模块融合，整合教学管理、学生后勤服务及系统底层运维体系，为师生提供便捷化的一站式线上办事途径，同时为系统管理者提供高效互通的数据化运营手段与追溯能力。

## 2. 系统概述
一体化智慧校园系统采用规范的前后端分离架构方案进行搭建，旨在实现校园基础业务的一体化与操作的便捷化。系统以角色化流程切入，保障三大核心模块群（教学执行、后勤服务、数据监控）业务闭环。系统核心目标为在有限的8周开发周期内提供一个功能清晰、可落地运行的智慧管理生态，为全校基础运转提供信息化保障。

## 3. 角色分析
本系统涵盖三类核心业务角色，各角色职能及权限边界清晰隔离：
*   **管理员**：系统的运行维护者与全局统筹者，拥有最高操作权限。主要职责包括系统底层基础数据（用户、字典项、角色）的维护、全局系统功能（如全校性公告下发、成绩防篡改终审、后勤报修分配等）的把控，以及运用宏观数据大屏监督全校运行状态。
*   **教师**：一线教学任务执行者。核心职责主要涵括各项教学业务的推进，例如考勤批次的发布与结果验收、学生各科成绩的严谨录入核对、个人授课资料与课件的共享维护、以及参与学生异常流转业务审批（如请假审核、考勤补签调整等）。
*   **学生**：平台核心业务受众。系统功能深度围绕学生在校学习与生活诉求展开，能够让学生高效完成周课表查询、在线考勤一键签到、电子请假申请办理、科目成绩查询申诉、图文宿舍报修提交，并实时跟踪个人校园卡与图书借阅状况。

## 4. 业务需求分析
### 4.1 登录与权限模块
*   **目标**：保障系统数据访问的安全准入边界，并确保各角色业务界限安全隔离。
*   **业务流程**：用户使用账号密码及动态图形验证码发起校验 -> 后端比对验证码并使用单向散列算法比对密码凭证 -> 校验通过后派发系统Token -> 前端解析Token与角色信息，动态渲染专属导航菜单树 -> 用户在分配的权限范围内开展业务操作。
*   **功能点**：账号密码多体系登录、防恶意登录验证机制、鉴权Token超时与刷新管理、API级多层越权防护、连续失误防爆破锁定、个人基本面资料管理与密码修改。
*   **业务规则**：密码强制使用不可逆单向加密标准固化存储；平台操作菜单纵向下探不超过3级，按三类不同身份强隔离；系统闲置状态满2小时Token自动销毁登出。
*   **异常情况**：若任一账号连续遭遇3次错误密码输入记录，系统将自动冻结拦截1小时，需管理员手动介入排查并解锁或等待自然超时失效。

### 4.2 系统管理模块
*   **目标**：提供安全溯源的基础框架，维持底层系统数据的标准化与一致性运作。
*   **业务流程**：管理员进入用户管理体系 -> 执行人员信息的增加或借助标准Excel模板进行批量入录与分发 -> 后端检验数据合法性并自动映射系统账号 -> 针对应用字典及菜单架构，通过可视化界面配置落表同步触发全局系统应用缓存清洗。
*   **功能点**：立体化用户管理（涵盖搜、改、启停与批处理机制）、参数化字典台账维护、菜单可视度及角色授权树制定、多链路日志（核心操作与账号登入事件）监控回溯。
*   **业务规则**：系统各用户账号（学号/工号）字段具有强制唯一约束限制；字典项及菜单架构发生关键变更后，必须保障多端缓存池的一致性同步革新。
*   **异常情况**：若导入配置的Excel表格存在非法空字段或与现存学号起冲突的情况，系统阻断导入写入并将异常行号连同错误详情抛入提示面板，执行该行数据级别的回滚。

### 4.3 课程管理模块
*   **目标**：搭建全方位的教与学业务关联，推动教学信息与电子资料的高度共享化。
*   **业务流程**：管理员建立标准课程档案并下放关联到对应行政班级及执教教师 -> 教师可基于名下辖管课程上传并管理各类电子教案 -> 学生通过日视图与周视图交替课表精准定位于上课预告，可实时下载查阅课件，亦能对结课项目发起量化双维评价。
*   **功能点**：教务级课程基础资料统筹规划、师生及班级的课程映射树匹配、多时间跨度课表动态渲染、教学文件全生命周期管控、星级指标与文字约束关联课程评价系统。
*   **业务规则**：课程编码强制作为实体的唯一索引码不得复用更替；所有文字课程评价提交上限封顶为200字；所有上传教辅资料单体体积不可跨越10MB物理限制标准。
*   **异常情况**：若教师上传格式不在此次放行的安全白名单（严控至文档/图片），或由于体积过载触发预警，流程立即中止响应并在前端展示容量与格式拦截红牌提示。

### 4.4 考勤管理模块
*   **目标**：彻底革新传统纸质喊名制度，构筑电子化的签到防伪底座与请假自动流转审批中枢。
*   **业务流程**：教师指定生效班级并在某课程范畴确立特定考核时段（预设常规30分钟基准）下达考勤任务 -> 学生基于免定位免扫码环境执行秒级远程签到指令；遇不可抗拒出勤障碍则切换发起电子请假工单 -> 授教方流转审核该请求 -> 时效截止，系统智能封薄并出具考勤台账数据报表 -> 后端预留极端场景下的手动出勤纠偏流或申诉通道至更高管理员复核。
*   **功能点**：动态打卡任务投递下发、一键式实时考勤签到板、师生连线的双向请假信箱与审批回执、班组出勤率宏观报表输出工具。
*   **业务规则**：凡越过签到截止红线且缺失合法获批请假背书的账号均标记记录为缺勤状态；合乎规流程的请假生效单将自动接管覆盖当前特定时段的原始状态。
*   **异常情况**：若由于不可控网络延迟甚至客户端卡顿致其遗漏报备，教师坐拥后台单一数据特例考勤特批修正权；若师生交涉遇到处理死角或认定不公，记录挂起转交由行政总管角色进行终极仲裁干预。

### 4.5 成绩管理模块
*   **目标**：规范校内阶段考核打分及内审监察环节，打造多级防篡改审核高墙与全维度可探源的透明成绩追踪体系。
*   **业务流程**：期末成绩清算打响，教师调取专属班级空白模板通过脱机核分后批量灌入模块 -> 数值精准通过无遗漏后，发起提请结案流程进入待核准状态 -> 二级管理员进行系统复审及抽查查勘；如符合常规发文审核通过 -> 冻结该批次状态 -> 解禁学生端对个人考量数据、横向校内排行的公开查询权域并留存二次异议申请入口。
*   **功能点**：分数灵活量规支撑（兼容百分制乃至特定等级制）录入操作、防篡改闭环复议流程、学子端自助排名追踪揭榜、课程及学阶的及格率中位数多元统测体系。
*   **业务规则**：每笔成绩池一经推入送审流段，底层修改入口立刻焊死查封阻挡一切私自修订动作；唯有经历督导层面退回（被驳回且详细随带核正理由批注）后，该锁区方能破冰重启修订通路。
*   **异常情况**：若系统接收的数据流包含跌破0分基准录入或飞出正常统算数值域（非0-100区间内）直接予以拦阻打单并标记单元格报错；如学生启动复查申诉，工单必须同步外挂佐证图像凭照才能流进后方校调复核程序中。

### 4.6 公告通知模块
*   **目标**：保障校园内重大事件以及行政决议快速、定点、无盲区地推送到指定利益相关体。
*   **业务流程**：管理方撰写兼备富文本排版规格的消息蓝本 -> 于发送端切分限定指定投喂群众标签（精准锁定特定角色体系甚至深入细化某一核心行政班号）发号施令 -> 系统针对中选目标客户端自动悬挂未读标志红点 -> 最终用户阅读消费对应资源并清除标志池数据留存归档。
*   **功能点**：富文本校政发布控制台、按组群切割精确触达策略引擎、状态反馈交互红气泡面板、重要备忘文档长效收藏模块。
*   **业务规则**：发布端掌控最高审阅特权可无死角巡查各个发文批次底部的触达效率分布图及未消费人员名册。
*   **异常情况**：当系统试图往已经被列入黑名单抑或是清退结业标志的死库账号中拉动最新公告通知推送时，触须应被底层判断自动识别切断并释放投寄资源空耗浪费。

### 4.7 宿舍与报修模块
*   **目标**：有效提供静态后勤宿舍资源池的分配重组工具，同时响应实时的床位基础设施修缮诉求闭环。
*   **业务流程**：以数据手段主导新生群体的物理入驻床位分派 -> 日常生活中由于基建设施磨损引发学生附带灾损实地照相的发文申告，同时根据灾损强度评断危机级别 -> 宿管职员跟进将抢修事项拆转分包更新维护脉络（初始挂单 -> 奔赴处理 -> 清扫归置闭环完成） -> 结果反推回显端后触动使用者开启最终核验计分打分流程。
*   **功能点**：涵盖横向建筑楼宇与纵向房间号的矩阵铺设与宿客入注管理盘面、图文并茂智能化修缮报废报修单投递追踪机制、工单流转动态呈现链、多维完工满意度评核体系。
*   **业务规则**：任何修缮申请应无条件携带损坏区域即时拍照作为前置证据入库；在现有同类型问题尚在“悬而未决”链路节点期间系统硬性锁定遏止其重新叠加发单重复报警举动；整条作业流只由管理总台做最后竣工勾查并撰写验收结语才得圆满核销掉。
*   **异常情况**：若处于高峰招生期遇到资源被抽干却依然执意指派新生入驻饱和空间的情况，系统需抛出严重冲突异常阻绝不合理请求通过。假使某些严重且火爆的报文挂靠在板单超过临界周期尚处于停摆状态未有人响应动作情况，应利用颜色代码在管理端闪烁高亮介入报警。

### 4.8 校园卡与图书借阅模块
*   **目标**：将实态物质财产（校园通IC支付账户及藏馆图册资源库）同线上虚拟履历精准映射对应。
*   **业务流程（校园卡）**：每一次校园外部刷挂发生真实金融清算皆将事件副本抛给中心系统存储记忆 -> 学生随时连线抽查个人的多时点和多地域破费花销记录动向；卡片不慎失落后在线求援至内勤系统施以资产保护性冻结。
*   **业务流程（图书借阅）**：藏书处把图书名目载录信息架构至系统内 -> 学生端启动检索搜寻并发起电子借阅动作过境 -> 如持有在手时段超出规定约定限制后被智能系统列上黑榜并频繁鞭策催归。
*   **功能点**：学生版校通卡明细查询功能及管理员端宏观流水监控和挂失处理室模块；包含典藏扩容补货登记、学生搜索借取库和失期自动归集警告处理流程等。
*   **业务规则**：有关卡体资金盘面功能限定只能够调取外域接口执行读取流水业务逻辑而封死所有企图在本域系统内部私造金额账务及人为发起充数的后门；只要借调所需图册超出当前实有冗余库房藏本便无条件拦截操作。
*   **异常情况**：如果一名学生违约霸书不还累积滞纳严重，智能判断中心应自动锁定限制他在任何未来时期内新一轮藏馆资源的征用权利乃至清账抹平完毕。

### 4.9 数据统计模块
*   **目标**：采用多维度、高度定制的统合驾驶舱，沉淀全维度杂乱微观数字为有形的大视角图形帮助反哺各项校园行政定策和私人查缺补漏。
*   **业务流程**：源源不断的流水模块基础源头每时每分都流入大汇合仓集中计算池 -> 以快照或者直接取数的形式映射组装出适配当前查看访问者属性图表 -> 并最终拆分成曲线走势等形态同时整合个人亟待破局处理的跳转急道于登录大盘前端内展示。
*   **功能点**：俯瞰全视角的超级数据透视展示面板（面对于高层管理员）、剖析所属教学班组细化质量评估的小微版讲台大屏（为教职工作者使用）、和涵盖各科目考出勤概貌兼顾督催自我待办事宜日程的私人仪表。
*   **业务规则**：面板内呈现的一切统筹因子在抛出前端前必须要依托查阅者的越权边界防护罩施行最高严密级别的清洗过滤规避敏感泄密发生；凡是待办事项板块等需加急特级任务须强势置入画面核心地带不允许推移折叠。
*   **异常情况**：假定恰逢全效期末核查使得大批量账号在瞬时涌入查看繁琐宏大数据时极可能诱爆实时算力负载临界峰值时系统有机制从内存库中倒回前一段时间最近更新的安全周期切片快照确保呈现绝不至于抛白死机报错阻断运转逻辑。

## 5. 功能需求清单
基于业务诉求梳理，以下功能需求清单以结构化视图规范展现系统的完整落地图谱：

### 5.1 基础综合平台
*   **登录与安全管控中心**：复合式凭证校验防线、滑转式图文双防爆破阻滞网套件、超时踢出系统回收池逻辑。
*   **个人信息与权限枢纽中心**：资料修正与隐私维护工具区、底层树状阶层角权编织与发放台、系统与人互通互动的高透光消息分拣提醒站内信中心。
*   **系统与资源配置内核**：账号名录增拓删削核心库及标准化Excel对导入库模块、具备全链路同步缓存刷新的字典大盘基站及多向条件抓取多层检索的全方位行为记录审记监听中芯。

### 5.2 核心教研生态子库
*   **课程统御管控带**：课程全息建档入图体系及灵活跨界连带挂载班系分放引擎、全视角按需变幻式课表投射功能仪、电子数字讲义共享下储中转交互站、基于实名或半匿名性质课程回评双规互评器。
*   **动态考勤雷达网络**：自定义时空条件的电子布道场建立机能、前端设备解绑自由签入通道、横穿师生界限审批的请假申请工作票核定链路及涵盖广域和狭义空间的防逃课纠错宏观出勤汇总工具集。
*   **成绩质量流转考核仓**：依托线下离接数据模型批量贯通接口线、强硬固化的成绩多层卡口锁扣拦截与终核审批闸门、支持个人和班级及宏观三向解剖查阅及分数统计分析报表平台和防冤案电子申查复议机制系统。

### 5.3 师生关怀服务集市
*   **信息及物料调度中枢**：全方位精准打击不遗漏定向富文本快讯弹窗及校令宣贯站台、涵盖床位空间立体配比调度及智能动态防患排查入住管理的宿舍行政指挥库、附带实地留痕的图像图文并用宿舍工程急救修缮呼叫抢通单及处理结调中台。
*   **校园数字孪生镜像中心**：单向镜像外接金融流通系统的财务溯流核算账本以及快速触发灾损防线的保护性卡片挂失操作、实体书库图册管理档案更新台及线上超限待办警报和借款催缴预告仪体系。

## 6. 业务流程分析
以典型场景“学生成绩审核与查询闭环流程”为例做业务核心流程解构，展示多岗位角色如何协同运转达成规范化闭环作业：
1. **录入与校验阶段**：期末考核结束后，一线教师凭借自身权限登录体系中心，并下载下发的专属班级的预制模板成绩空表，进行离散脱机批阅并填充数字录入直至完成，之后执行动作上传；后端服务侦听上收资料开始横扫单元格按照特定设定的分数区划尺度启动容忍校验法则阻绝一切粗制滥造坏数据流冲入基础库。
2. **提单提审阶段**：机器验证绿灯亮通并交还回传提示后，教师执行最后确认动作敲击落定递呈按钮；此瞬间所辖涵盖整批分数组皆悉数转为加密锁定状彻底禁止倒退撤回更改操作，流程接力棒随即脱离教导手口滑向了上层最高督办行政核验端池内。
3. **二度复查裁决阶段**：最高级别管理员切入后勤复校大局控制面板鸟瞰整个全局盘子情况，如明察秋毫发现存在违背规则分段畸形等可疑蛛丝马迹则有权立刻叫停，利用驳回意见通道写明原因打回初始态交由相关讲师回炉精进纠改；一旦查探通体完美则爽快点击拍板盖章通过并将其永远入归建历史档永久封存。
4. **公告天下与排忧解难阶段**：经历终极审批通过发令起效的刹那一刻起，平台则在第一时间即时撕开以往处于封阻隐蔽下的暗纱层全面解放，面向所有的学子终端同步推送其最终成绩和多维名次报告；顺理推章的，若有个别人士针对于考核参数存有的深层疑点和不满皆可利用挂载上随同电子凭据影像或凭据快照提发在线伸冤追讨诉讼通道至校方的调查处理重巡特批链路求得完美核处了断整个年度事件终点。

系统的稳定高效流转依赖底层极其稳固规范和严格同构标准下的标准数据实体模型为支撑背板，各类关键结构库群需服从以下基础切口分布的建设大纲法则准绳：
